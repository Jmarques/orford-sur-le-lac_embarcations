Où la dérivation est profonde,
mais la présentation et les I/O n'ont pas de foyer

Le fil rouge de toute la revue en une phrase : la dérivation métier est centralisée exactement comme le domaine l'exige (statut, état de demande, hors-quota — dérivés, jamais stockés, une seule fois dans grille.js) — mais la couche qui met ces dérivations en mots, en couleurs et en requêtes réseau existe en 3 à 6 copies qui chevauchent les seams des modules.

module logique dupliquée seam (interface) fuite à travers le seam module profond

Six candidats, classés. Chacun est un deepening : plus de comportement derrière une interface plus petite, un seul foyer à tester. Vocabulaire : module · interface · profond/superficiel · seam · adapter · leverage · locality.

Strong in-process · nouveau seam 01

Un module client backend qui n'existe pas

site/fiche.js · fiche-adresse.js · fiche-demande.js · a-traiter.html · structures.html · index.html

Avant — la complexité est déjà étalée

envoyerAction copié 3× · class ErreurApi déclarée 6× · reponseAcceptee (l'enveloppe ok / erreur / accesRefuse) ré-écrite 5× · 11 sites fetch.

Après — un module profond concentre tout

Une seule interface : poster(action, corps) → résultat | ErreurApi. L'astuce text/plain (pas de preflight), le mot de passe dans le corps (déc. 0008) et l'enveloppe vivent une fois.

Problème

« Comment parler au backend et lire un refus » est un invariant du système, recopié dans chaque appelant.

Solution

Un module client — poster(action, corps) — porte l'URL, l'enveloppe et ErreurApi ; les appelants passent action + corps.

  • locality : un nouveau code de refus = 1 édition, pas 6
  • leverage : 1 interface, 11 sites d'appel
  • supprime 6× ErreurApi + 3× envoyerAction
  • premier module frontend testable en node (fetch injecté)

Strong exposer des helpers privés 03

Les helpers profonds sont déjà écrits — mais privés

grille.js (dateLisible_, chercherMembreParCle_) · fiche.js · fiche-adresse.js · fiche-demande.js · a-traiter.html

Chaque règle vit une fois dans grille.js (souvent en privé) et en N copies au-dessus du seam

Après : chaque règle = une barre exportée. L'invariant « première position gagne pour un numéro en double » et l'encodage mailto (XSS) cessent de pouvoir diverger silencieusement.

Problème

La version canonique existe mais reste privée (_), donc chaque renderer la ré-écrit — et les copies peuvent diverger.

Solution

Exposer dateLisible / chercherMembre, ajouter formatAdresse, positionParNumero, hrefEcrire au module partagé.

  • faible risque : additif, une passe
  • locality : 19 copies → 5 foyers
  • l'invariant « 1ʳᵉ position gagne » cesse d'être re-décidé
  • l'encodage mailto (XSS) vit une fois

⚠ Touche les décisions 0004 / 0009

Le module unique copié tel quel est un choix délibéré (pas de build). Rouvrir seulement si la navigabilité (31 exports, 7 concerns) pèse assez : le copieur apprend à concaténer plusieurs fichiers au lieu d'un copyFile. Le seam de copie et son test d'égalité survivent.

La navigabilité IA est le but explicite de cette revue — d'où « worth exploring » malgré l'ADR. Ce n'est pas urgent : rien n'est cassé, c'est du confort de lecture et de locality.

  • locality : 1 concern = 1 fichier
  • un import ciblé au lieu de 31 globales
  • coût : le copieur devient un concat + garde-fou
  • déplace la complexité, ne la concentre pas — d'où le rang

La fiche fuit par son options et par un id DOM

Pour tracer un geste utilisateur on rebondit entre 3 fichiers : contrat donnees() implicite et non-uniforme (fiche-demande veut aussi demandes), page qui atteint la fiche par id DOM en dur, chorégraphie de navigation à 3 drapeaux répartie sur page + fiche.js + fiche-adresse.js.

Problème

Tout ce qu'un appelant doit savoir (clés de donnees, ordre du re-render, id du drawer, retour de navigation) est presque aussi gros que ce que la fiche fait — interface superficielle.

Solution

Interface explicite : la fiche reçoit ses données (pas une closure sur l'état de page), émet des événements (geste, retour) ; la page ne connaît plus son id DOM.

  • la chorégraphie de navigation vit en un lieu
  • le focus du drawer cesse d'être écrit en double
  • plus gros chantier → « worth exploring », pas « strong »
  • dépend en partie du candidat 03 (helpers partagés)
Speculative locality dans la coquille impure 06

Backend : majLigneParCle_ écrit deux fois la même boucle

apps-script/sheets.js:69 (majLigneParCle_) · sheets.js:251 (upsertMembre_) · Code.js:31

Avant — même find-merge-write, deux fois

majLigneParCle_(cle, valeur)

getDataRange → boucle → fusionnerLigne → setValues · match = cellule exacte

upsertMembre_(membre)

getDataRange → boucle → fusionnerLigne → setValues · match = cleAdresse (normalisé)

La seule différence est le prédicat de correspondance. La contrainte « n'écrire que les colonnes fournies » (déc. 0012) est donc codée deux fois.

Après — prédicat paramétré

majLigne(feuille, correspond, objet)

une boucle find-merge-write ; le prédicat correspond(ligne) varie

upsert membre = majLigne(…, ligne ⇒ cleAdresse(ligne)===cle, …)

Bonus voisin : Code.js:31-40 ré-liste les 5 clés de lireInventaire() alors que les 8 autres handlers transmettent la valeur opaquement — pure redondance à retirer.

  • petit gain de locality dans la glue impure
  • la règle « écrire par nom de colonne » = un foyer
  • reste non-testable en node (SpreadsheetApp) — d'où « speculative »
  • gain modeste : ~15 lignes, à faire en passant

Par où commencer

Recommandation

01 — Le module client backend

Le seam manquant le plus net : 11 sites fetch, ErreurApi déclarée 6×, envoyerAction copié 3×, l'enveloppe ré-écrite 5×. Purement additif — aucun ADR touché, aucun build à changer. Et il ouvre le premier module frontend testable en node (fetch injecté) : aujourd'hui toute la couche fetch/enveloppe a zéro couverture unitaire. Meilleur leverage pour le moindre risque.

Ensuite 02 puis 03 — même thème (« la dérivation existe, la présentation n'a pas de foyer »), même faible risque. 04 et 05 sont des chantiers de fond à décider séparément ; 06 se fait en passant.

Dites-moi lequel vous voulez explorer — j'enchaîne alors sur /grilling pour dérouler l'arbre de conception (contraintes, forme du module approfondi, ce qui vit derrière le seam, quels tests survivent).
